Format: `ADR-NNNN · date · title` → Context / Decision / Consequences. Append-only; superseding requires a new ADR referencing the old one.

## ADR-0001 · 2026-07-10 · CAD kernel = OpenCascade.js in a Web Worker
**Context:** Need B-rep booleans, fillets, chamfers, and STL meshing with deflection control, fully client-side, GitHub Pages compatible.
**Decision:** OpenCascade.js (OCCT → WASM), custom trimmed single-threaded build, isolated in a dedicated Web Worker; alternatives rejected: JSCAD/manifold (mesh-based, no B-rep fillets), replicad (less control over topology and build size).
**Consequences:** ~10–15 MB one-time WASM download; manual WASM memory management (`.delete()` discipline, live-handle counter); all kernel access via typed message protocol.

## ADR-0002 · 2026-07-10 · Option B: solver-free sketching (numeric input + snapping)
**Context:** Full Fusion-style constraint solver (planegcs) estimated at 30–40% of total project effort; target users (3D-printing hobbyists) need precision and speed, not design-intent propagation. Shapr3D-style input UX identified as the desired feel.
**Decision:** v1 ships without a constraint solver. Precision via SnapEngine (point snaps + inference guides) and NumericInputMachine (typed values at creation, properties panel for edits). Committed geometry is baked; no propagation on edit.
**Consequences:** ~30% effort reduction; editing one entity does not update related entities (documented in UI). Mitigation: sketch schema is constraint-ready (C6 — reserved `constraints`/`dimensions` arrays, stable point roles), making a v2 planegcs integration additive with no schema migration.

## ADR-0003 · 2026-07-10 · Edge references via geometric fingerprints (no full topological naming)
**Context:** Robust topological naming across parametric edits is a multi-year problem (cf. FreeCAD history).
**Decision:** Fillet/Chamfer/Combine reference edges by fingerprint (midpoint + direction + adjacent face kinds, tolerance-matched at regen); sketch-on-face planes reference faces by fingerprint (centroid + normal + area) with a plane snapshot stored for main-thread editing. Failure to resolve → op error state, user re-picks. Same philosophy extended to profile identity: entity-set hashes, never detection indices (ARCHITECTURE R7a).
**Consequences:** Occasional re-selection after heavy upstream edits; drastically simpler kernel bookkeeping; honest limitation surfaced in UI copy.

## ADR-0004 · 2026-07-10 · Layered architecture with CI-enforced dependency rules
**Context:** Multi-month solo/AI-assisted project; highest risks are layer erosion, logic duplication, and doc/code divergence.
**Decision:** Layer catalog + dependency-cruiser gate (ARCHITECTURE §3), single write path via CommandBus (§4), per-operation three-file registry pattern with completeness test (§7), docs-first change policy (§15).
**Consequences:** Slightly more ceremony per feature; structural violations fail CI instead of accumulating; adding an operation is a bounded, predictable change set.

## ADR-0005 · 2026-07-10 · Copy/Paste is parametric-positional (revised during plan critique)
**Context:** Original wording ("baked snapshot, source edits never propagate") contradicted C5 — XML persistence is parametric replay, so a shape snapshot cannot be serialized, and replay makes upstream-edit propagation unavoidable.
**Decision:** `CopyBody` reproduces the source body as of the op's timeline position. Edits to ops earlier in the timeline propagate into the copy on regen (Fusion-like); ops appended after the copy do not.
**Consequences:** Fully C5-consistent and replayable; behavior explained once in UI on first paste. Truly frozen copies would require a separate "bake to mesh body" feature — out of scope v1.

## ADR-0006 · 2026-07-10 · Corrections from pre-implementation plan critique
**Context:** Structured critique of v1.1 documents before any code found: index-based profile references (silent wrong-geometry risk), missing hole support in profiles, sketch-on-face contradiction with R7, approximate measure radii, dependency-cruiser config not matching the layer table, undo/redo regen path unspecified.
**Decision:** Entity-set-hash profile identity (R7a); Profile = outer + inner loops with MakeFace holes; face fingerprints + plane snapshots for sketch-on-face; exact edge metadata in mesh transfer (R5a); config rules aligned with layer table incl. worker-entry-only exception; undo/redo emits standard dirty-marks (R2).
**Consequences:** No silent geometry substitution; canonical hole workflow specified from day one; enforcement config is authoritative alongside the table (both change together, per rule comment in config).

## ADR-0007 · 2026-07-10 · Corrections from plan critique round 2
**Context:** Second structured critique after Option-B corrections found: regen pseudocode assumed a linear single-body chain (wrong suppression semantics for a multi-body DAG, undefined executor ctx); XML loader accepted newer minor versions (guaranteed silent data loss vs the planned v2 constraints minor bump); modeling-while-rolled-back undefined; evaluated body set had no state owner; multi-tab autosave key collision; Revolve axis source unbounded; i18n mechanism and Pages base path unspecified/hardcoded.
**Decision:** BodyStateMap execution context with per-op deltas and per-op-class suppression semantics + 'skipped' status (ARCHITECTURE §9); reject any newer schema version (§11); Fusion-style insert-at-marker for rolled-back modeling (F1); evaluated body set owned by kernel client, derived state (§5); per-tab autosave keys with newest-first restore (§7); Revolve axis restricted to same-sketch line or origin axis (F3); in-house minimal t() with flat EN catalog; Pages base single-sourced from repository name.
**Consequences:** Worker implementation in M3 starts from correct multi-body semantics; forward-compatibility data loss impossible by construction; rollback editing behavior matches Fusion expectations (C4).

## ADR-0008 · 2026-07-10 · Corrections from plan critique round 3
**Context:** Third critique found the XML schema baked per-entity coordinates while F2/CLAUDE mandated shared point references — round-trip would silently destroy sketch connectivity and undermine C6 (v2 coincidence needs point identity). Also: Line angle reference undefined (untestable M2 acceptance), no protocol support for F6 triangle-count preview, M2 acceptance not exercising holes.
**Decision:** Per-sketch point pool in schema; entities reference `PointId`s; role refs are accessors. Face-based sketches serialize fingerprint + plane snapshot. Line angle absolute to sketch +X, relative field added for chained segments. `meshStats` request kind (counts only, no buffers). M2 acceptance extended with plate-with-hole and round-trip connectivity check.
**Consequences:** Sketch topology is identical before and after save/load; solver upgrade path intact; STL dialog preview has a real, cheap protocol path.

## ADR-0009 · 2026-07-10 · No license — all rights reserved (supersedes MIT proposal in ADR log context)
**Context:** Owner decided the repository stays public on GitHub but is for personal use only at this stage.
**Decision:** No LICENSE file, no license headers, no open-source badges. Default copyright applies: all rights reserved. README carries a copyright notice. Vision wording changed from "open-source" to "source-available".
**Consequences:** Under GitHub ToS, others may view and fork the public repo, but receive no legal right to use, modify, or redistribute the code. External contributions should not be accepted (no grant basis for PRs). A license can be added later at any time — the reverse (retracting a grant) is not possible, so starting restrictive is the conservative choice.

## ADR-0010 · 2026-07-10 · M0 scaffold judgment calls
**Context:** Building the M0 scaffold (Vite+React+TS, viewport, CI gate) surfaced several small decisions not fully pinned down by the three binding documents.
**Decision:**
- **Barlow font not bundled in M0.** `tokens.css` declares `--font-heading`/`--font-body` as `'Barlow', -apple-system, 'Segoe UI', Roboto, sans-serif` but ships no webfont files or CDN `<link>` — a Google Fonts fetch would be a "server call" on every load, in tension with C1/C2's static-host, offline-friendly intent. The real Barlow files land at M7 (styling milestone) with the token values unchanged; system fonts stand in until then.
- **HUD text crosses layer boundaries via props, not direct `t()` imports.** `viewport/Viewport.tsx` cannot import `app/i18n` (dependency-cruiser `viewport-scope` correctly rejected this on first attempt). Established convention: `app/` resolves `t('key')` and passes the translated string down as a prop to any `viewport/`-owned HUD element. Applies to all future overlay/HUD text in `viewport/`.
- **ESLint flat config (`eslint.config.js`) with `typescript-eslint` typed linting**, including `switch-exhaustiveness-check` (enforces CLAUDE.md's "exhaustive switch with never checks") and `no-explicit-any`. No legacy `.eslintrc` — flat config is the current ESLint default and CLAUDE.md doesn't pin a config format.
- **stylelint's `no-hardcoded-color` rule implemented directly** (`color-no-hex` + `color-named: never` + `function-disallowed-list` for `rgb/rgba/hsl/hsla`, overridden only for `app/ui-tokens/tokens.css`) instead of pulling in `stylelint-config-standard`, keeping devDependencies limited to the fixed stack.
- **Orbit scheme left at Three.js `OrbitControls` defaults** (LMB rotate / MMB dolly / RMB pan). MASTER_DOCUMENT §13 lists "default orbit scheme" as an explicit open decision (Fusion middle-drag vs. Shapr3D-style) — M0 does not resolve it, just needs *an* orbit control.
- `package.json` `license` field set to `"UNLICENSED"` (npm's standard "no license granted" marker), replacing `npm init`'s auto-generated `"ISC"` — consistent with ADR-0009, not a new grant.
**Consequences:** No network calls introduced at runtime; layer rules stay enforceable by dependency-cruiser without exceptions; tooling stays inside the fixed stack; orbit scheme and PL catalog remain explicitly open for a later ADR when MASTER_DOCUMENT §13 is resolved.

## ADR-0011 · 2026-07-10 · M1 OCCT binary, package version, and upstream typing gaps
**Context:** M1 needed a working OCCT WASM kernel today. ADR-0001 committed to a "custom trimmed single-threaded build" (~10-15MB estimated); actually producing one means fetching OCCT source + the Emscripten SDK and running a from-scratch C++→WASM build (multi-GB downloads, 30-90+ min, its own Docker toolchain) — out of reach for this session. Separately, the `opencascade.js` npm package's `latest` tag (1.1.1) ships **zero** TypeScript type declarations, incompatible with CLAUDE.md's "no `any` in src/" — hand-authoring bindings for a ~200k-line generated API surface is not viable either.
**Decision:**
- **Ship the full, untrimmed OCCT WASM binary for M1** (owner-approved explicitly, this session). `tools/occt-build/` remains the tracked home for the real custom trimmed build — a later, separate effort that drops in as a same-path replacement in `public/wasm/` with zero code changes on our side.
- **Use `opencascade.js@2.0.0-beta.b5ff984`, not the `1.1.1` "latest" tag.** Only the beta ships a generated `.d.ts` (`dist/opencascade.full.d.ts`, ~200k lines) alongside the WASM (`dist/opencascade.full.wasm`, ~50MB — smaller than 1.1.1's untyped 66MB build, a bonus not the reason). Confirmed single-threaded (no `pthread`/`SharedArrayBuffer` references), satisfying C2.
- **WASM assets live in `public/wasm/`** (copied from the npm package once, not imported from `node_modules` at runtime): `kernel-worker/occt.ts` loads `opencascade.full.js` via a runtime `import(/* @vite-ignore */ ...)` against `${import.meta.env.BASE_URL}wasm/...`, with `locateFile` pointed at the co-located `.wasm`. This keeps the worker's own bundle small (~3KB), never hardcodes the Pages base path, and means the eventual trimmed-build artifacts replace these two files with no loader changes.
- **Two upstream generation gaps in the beta's `.d.ts`, bridged in `kernel-worker/occtCompat.ts`** (never with bare `any`):
  1. Enum "type" aliases (`TopAbs_ShapeEnum`, `TopAbs_Orientation`, `Poly_MeshPurpose`) model the whole runtime namespace object, not one member's type — `enumMember()`/`enumArg()` cast through `unknown` at the handful of call sites that touch enums.
  2. `Graphic3d_ZLayerId` (used as the return type of nearly every OCCT count/index method — `NbNodes()`, `NbTriangles()`, `Triangle()` indices, ...) is referenced constantly but never itself declared anywhere in the generated file. A `declare module 'opencascade.js' { export type Graphic3d_ZLayerId = number }` augmentation fixes this for `tsc -p` cleanly, but — confirmed empirically via isolated repro — does **not** reliably reach typescript-eslint's `projectService` diagnostics for a file that also imports real (non-augmented) types from the same huge module, a caching/ordering quirk in that tooling, not something fixable from our side. `int()` (same `unknown`-cast shape as `enumArg`) wraps every such value at its call site instead.
- **`KernelClient`/`kernel-worker` implement only the M1-necessary slice of `protocol.ts`**: `init` / `tessellate` / `exportStl` / `dispose` / `stats`. `regen` + `RegenPlan` arrive in M3 with `document/ops` and `services/RegenScheduler` (nothing to type them against yet); `progress` arrives alongside the multi-op regen loop it reports on; `meshStats` arrives with the M6 STL dialog's triangle-count preview. `dispose` is new relative to ARCHITECTURE §6's example union — M1's stand-in for regen's cache-invalidation disposal (§9), and stays useful afterwards for explicit body deletion.
- `core/errors.ts`'s `KernelError.opId` changed from required to optional — M1's `init`/`stats`/`exportStl` failures have no associated op (no timeline exists yet); ARCHITECTURE §12's `KernelError { opId, code }` doesn't mandate `opId` on every instance.
**Consequences:** ~50MB one-time download stands until the real trimmed build lands (revisit bundle size then); LGPL-2.1 (`opencascade.js`/OCCT) redistributed as an unmodified binary loaded at runtime, consistent with ADR-0001's kernel choice; `occtCompat.ts` is now the one place future M2-M7 code routes OCCT enum/count values through, rather than scattering casts; M1's acceptance test (hardcoded box → tessellate → render → STL download, live-handle counter to baseline) verified in a real Chromium session, not just unit tests.

## ADR-0012 · 2026-07-10 · M2 sketch environment judgment calls
**Context:** Implementing F2/R7/R11 surfaced decisions the binding documents leave open.
**Decision:**
- **Profiles form through shared-endpoint topology only.** Loop detection is half-edge face traversal over entities meeting at shared pool points (plus standalone circles); entities that CROSS without a shared endpoint do not split regions in v1. Fusion does split at crossings; our M2 acceptance cases don't need it, and mid-entity splitting requires entity subdivision (a document mutation) that belongs with a deliberate design pass. Open contours, dangling tails, and bridge edges are flagged as open (F2 "allowed but flagged"). Corollary: tangent-continuous joints (line meeting tangent arc) have equal departure angles — the traversal order at such nodes is well-defined but arbitrary between the two; no known-wrong case, noted as a watch item.
- **Every closed loop is a selectable profile AND nested loops are their parent's holes** — a plate with a circular cutout yields the ring profile (inner loop = the circle) plus the disk profile, exactly like Fusion. The MASTER_DOCUMENT F2 phrasing "one profile with a hole" is satisfied by the ring; the disk is additional, not contradictory.
- **Keyboard-only tool starts default to the sketch origin.** Line/circle/polygon tools invoked purely via numeric input (no prior click) anchor at (0,0): F2 defines length/angle fields relative to a start point but no keyboard mechanism for placing the first one. Origin-start matches how brackets are actually drawn and makes the M2 "keyboard only" acceptance well-defined.
- **Commit-time coordinate merging** (tolerance 1e-6 mm) in `GeometryPlan`: a committed endpoint landing exactly on an existing pool point references that point instead of minting a twin. Cursor snapping already yields pool-point refs via `sourceRef`; coordinate merging covers keyboard-only closure (the bracket's last segment ends on the first segment's start without ever snapping). Without it, keyboard-drawn shapes would be topologically open (C6 violation in spirit).
- **Snap priorities rank inference guides BELOW `on-entity`** (endpoint 100 > intersection 90 > midpoint/center 85 > quadrant 80 > on-entity 40 > guide-intersection 38 > align 35 > parallel/perp/tangent 33 > grid 20). First implementation had guides above on-entity; its own unit tests showed alignment guides swallowing every on-curve snap near axis-aligned geometry. When the cursor is on real geometry the entity snap must win — it carries the sourceRef that committed topology needs. Guides are an empty-space aid (matches Fusion behavior).
- **Numeric HUD is a fixed bottom-center panel, not cursor-following** — display-only mirror of the NumericInputMachine; keystrokes flow through one global handler, never DOM inputs, so Tab/Enter/Esc semantics stay exactly the machine's. Cursor-following float is M7 polish.
- **Tangent guide covers "drawing FROM a point on a circle/arc"** (tangent continuation). Line-grazing-a-distant-circle tangency inference is out of M2 scope.
- **World is Z-up** (Fusion/CAD convention): model XY plane is the ground; camera `up` is (0,0,1). Entering a sketch animates the camera along the plane normal with the sketch +v axis as view-up; orbit is disabled during sketch editing (pan/zoom remain) — the overlay projects through the live camera and would survive orbit, but locking rotation matches the M2 UX and Shapr3D's sketch feel.
- **Undo/redo (Ctrl+Z/Ctrl+Y) enabled in M2.** R2 requires every command to produce an undoable Transaction anyway; the history stack (whole-sketch inverse patches, capacity 100 ≥ the required 50) and bus plumbing were needed for correctness, so the shortcut wiring is trivial. The M3 milestone still owns the timeline-integrated undo UX.
**Consequences:** Sketch subsystem is fully unit-tested without DOM (R11): 100+ tests across snap/input/profiles/commands/tools; profile detection semantics documented before Extrude consumes profiles in M3; region-splitting-at-crossings is an explicit, logged limitation rather than a silent one.

## ADR-0013 · 2026-07-11 · M3 timeline, regeneration, and Extrude/Revolve judgment calls
**Context:** Implementing F1/F3 (timeline + Extrude/Revolve, the four boolean operations, undo/redo, live regeneration) surfaced decisions the binding documents leave open, plus a few structural choices needed to keep the single-write-path (§4), registry (§7), and regeneration (§9) rules intact end to end.
**Decision:**
- **A Sketch is a timeline op.** `DocumentState` gains `ops: TimelineOp[]` + `rollbackIndex`; creating a sketch mints a `SketchOp` in the SAME transaction as the sketch (two patches, one undo). The Sketch op carries no geometry — the sketch data still lives in `DocumentState.sketches` — but it owns the sketch's position in history, its suppression, and (via `producesSketch`, below) the dirty-tracking edge for sketch edits. Matches Fusion's timeline where the sketch is the first feature.
- **Four registries, kept in lockstep by a completeness test.** Beyond ARCHITECTURE §7's three (document `OP_DEFINITIONS`, worker `OP_EXECUTORS`, app `OP_FEATURES`), M3 adds a fourth main-thread registry — `services/regenPlan.ts` `OP_PLAN_RESOLVERS` — because plan construction (profile + revolve-axis resolution into world space) is op-specific but cannot live in `document/` (its output types `PlanProfile`/`WorldAxis` belong to `kernel/protocol`, which `document/` may not import). `registry-completeness.spec.ts` asserts all four cover `OP_TYPES` (R9). This is the honest reading of prime-directive #4 ("registry for every operation") given the layer rule.
- **`OpDependencies` gains `producesSketch`** (symmetric with `producesBodies`) so the regen planner marks a downstream op `inputsSuppressed` when its consumed Sketch op is suppressed, and dirty-tracking finds sketch edits — both WITHOUT any `if (op.type === 'Sketch')` switch (R4). No per-op type literal appears anywhere in `services/` or the timeline UI; op routing (dialog vs. re-enter-sketch) reads registry data.
- **Boolean ops take an explicit `targetBodyId`.** Join/Cut/Intersect name their target body in the op (validated: non-NewBody requires a target); the worker applies the tool to that body in the `BodyStateMap` and stores the result back under the target's id. NewBody mints a fresh `bodyId` at creation, stable across regens (§8). No implicit "last body" — the timeline is a multi-body DAG, not a linear chain.
- **Multiple selected profiles fuse into ONE body per op.** An Extrude/Revolve over several profiles builds each prism/solid and fuses them into a single tool shape before the boolean applies — one op → one produced body. Separate bodies = separate ops (Fusion parity).
- **Regeneration cancellation without message interleaving.** The worker bumps `latestGeneration` synchronously on each `regen` message arrival and yields to the message queue (macrotask `setTimeout(0)`) between ops, so a newer generation preempts an in-flight one at the next op boundary (superseded runs settle as a benign `StaleRegenError`, R6). Handlers are promise-chained (serialized) so a regen never observes a half-applied newer state.
- **Delta cache owns shapes; statuses are cached alongside deltas.** Per-op deltas store the shapes each op changed (§9); `freeFrom(k)` is the sole disposer (R8). An aborted regen can leave a shorter valid prefix than the next request's `fromIndex` assumes, so the loop clamps `fromIndex` to the cache length and re-reports cached statuses — no status is lost to a superseded generation. A boolean op's superseded target shape stays owned by the delta that produced it (replay-from-k needs it), not freed eagerly.
- **Timeline XML order is explicit.** The codec iterates the op registry (R10) and stamps each element with an `index` attribute; parsing regroups by tag (fast-xml-parser loses document order) and re-sorts on it. Order is timeline-significant, unlike sketch entities (which sort by id).
- **Dialog ghost-preview deferred to M7.** Extrude/Revolve dialogs commit on OK (validated form → `AddOp`/`EditOp`); live drag-preview of the pending solid is polish, not correctness, and depends on a preview-regen path the milestone doesn't need. Editing an existing op re-opens its dialog pre-filled; a Sketch chip re-enters the sketch environment.
- **Kernel is strictly downstream of the write path.** `app/store/kernelStore` holds only regen RESULTS (body meshes + per-op statuses); its sole writer is the `RegenScheduler` fan-out (R1). The worker + scheduler are created lazily on first app mount (`startRegen()`) so importing the store never spawns a Worker (tests/SSR). The M1 `kernelDemo` scaffold (which talked to the kernel directly, pre-document) is deleted; STL export moves to a toolbar button over the live bodies.
**Consequences:** The full edit→regen pipeline is proven end to end — a sketch-entity edit rebuilds the plan with the new profile geometry (deterministic services test), the worker builds the correct solid from that plan (executor golden tests: box, plate-with-hole = outer−hole, all four booleans, revolve washer, handle-count baseline R8), and the whole browser wiring runs in a real Chromium session (sketch → extrude → re-enter sketch → edit → live regen). 121 unit tests + 4 e2e green; lint/typecheck/depcheck clean. Region-splitting-at-crossings (ADR-0012) and dialog ghost-preview remain the logged open items.

## ADR-0014 · 2026-07-11 · M4 finishing ops + edge fingerprints
**Context:** M4 adds Fillet, Chamfer (F4) and Combine (F5). The defining decision is how a picked 3D edge is REFERENCED so a downstream fillet survives an upstream edit — the milestone acceptance ("filleted boolean part survives upstream edit or errors gracefully").
**Decision:**
- **Geometric edge fingerprints, resolved at regen (no topological naming in v1).** An `EdgeFingerprint { midpoint, direction, adjFaceKinds, tolMm }` is computed on the main-thread-visible body edges shipped by the worker (`tessellateBodyEdges`), stored in the op, and RE-RESOLVED against the input body at each regen. This is exactly ARCHITECTURE §8 line 160 / MASTER_DOCUMENT §4.2's accepted v1 tradeoff — persistent topological naming is a large subsystem deferred past v1.
- **Resolution = signature filter + nearest-midpoint within tolerance.** Candidate edges are scoped to those with the same sorted adjacent-face-kind signature (`plane`/`cylinder`/`cone`/`sphere`/`torus`/`freeform`) AND aligned direction (|cos θ| ≥ cos 5°, sign-normalized so an edge and its reverse share a fingerprint); the nearest midpoint wins if within `tolMm` (default 5 mm captured at pick). A modest dimensional edit re-resolves (midpoint moved < tol); a large one — or a removed edge — leaves no candidate in tolerance and `resolveEdges` throws `KernelExecError('EDGE_UNRESOLVED')`, which the regen loop turns into the op's `error` state with the last good state rendered. Both halves of "survives OR errors gracefully" are golden-tested.
- **Fillet/Chamfer modify a body in place; Combine consumes tools.** Fillet/Chamfer keep the target's `bodyId` (a feature ON the body, §9) and depend on it via `consumesBodies`. Combine (F5) is body-to-body Join/Cut/Intersect (never NewBody); it stores the result under the target id and, unless "Keep Tools", REMOVES the tool bodies from the map.
- **Delta cache extended with removals.** M3's delta recorded only shapes an op SET; Combine is the first op that removes an existing map body, so `OpDelta` gains a `removed: Set<BodyId>` (folded on replay: set changed, drop removed). Without it, a replayed combine would resurrect a consumed tool. Removed ids carry no owned shape — that shape stays owned by the delta that produced it.
- **Edge picking = viewport raycast on emitted edge polylines (F4).** The worker tessellates every body edge into a world-space polyline + fingerprint in `regenDone` (R5 Transferables). The viewport renders them as pickable `THREE.Line`s with hover highlight; a click toggles the edge into the session's pick set. A new services-side **plan-resolver** registry entry per finishing op keeps registry parity (now four registries × six OpTypes, R9); the finishing ops need no main-thread input resolution (edges resolve in the worker), so they share a no-op resolver.
- **Dialogs are non-modal during picking.** The Fillet/Chamfer dialog backdrop is `pointer-events: none` (only the panel captures clicks) and anchored top-right, so edge picks reach the viewport while the dialog is open. `EdgeFingerprint` is re-exported through the `kernel` barrel so `viewport/` can type the pick callback without importing `document/` (which the depcheck viewport-scope forbids).
**Consequences:** Fillet/Chamfer/Combine are spec-conformant with golden proofs (fillet/chamfer reduce volume; Combine Join/Cut/Intersect = 5000/3000/1000; Keep-Tools retains the tool; fingerprint re-resolves after a 20→21 mm edit and throws past tolerance) and the timeline codec round-trips all six OpTypes incl. edge fingerprints. The geometric-fingerprint limitation is explicit and surfaced as graceful op errors, not silent wrong-edge fillets; robust topological naming is the deferred follow-up.

## ADR-0015 · 2026-07-11 · M5 browser tree, copy/paste, measure, and 100-body performance
**Context:** M5 adds the browser tree (F8), whole-body Copy/Paste (F9), Measure (F10), and must keep a 100-body session usable at ≥ 30 fps.
**Decision:**
- **Body metadata is document state, lazily materialized.** `DocumentState.bodyMeta` holds `{ id, name, color, visible }` per body, keyed by the stable `BodyId` an op mints. A body with no entry reports `defaultBodyMeta` (id-as-name, brand teal, visible); rename/recolour/hide upsert one entry via `SetBodyName`/`SetBodyColor`/`SetBodyVisible` commands, each an undoable whole-list `BodyMetaPatch` (R2). Deleting a body in the tree deletes its PRODUCING op (`producesBodies`), with the same dependent-op warning as timeline delete — a body is not an independent entity, it is a feature's output (§9).
- **Copy/Paste is a `CopyBody` timeline op (parametric + positional).** Ctrl+C stores the selected body; Ctrl+V appends a `CopyBody` op at the rollback marker referencing that source + an optional XYZ translate. The worker copies the source AS IT STANDS in the BodyStateMap at the op's position (`BRepBuilderAPI_Transform`, copy=true), so edits to earlier ops flow into the copy while later ops do not (F9). The delta cache's new `removed` set (ADR-0014) already covers the map bookkeeping; the source stays owned by its producing delta. This is the seventh OpType — all four registries and the completeness test extend to it with no new machinery.
- **Measure snaps geometrically on the main thread.** Picks project the worker-emitted edge polylines' endpoints/midpoints to screen and take the nearest within a pixel tolerance, else the body-surface raycast hit; two points → distance + ΔX/ΔY/ΔZ, a single closed near-circular edge → radius/diameter (detected by low radius-variance about the polyline centroid, dropping the duplicated closing point). No kernel round-trip per pick.
- **Edge tessellation moved OFF the regen path (the key perf lever).** M4 emitted every body's pickable edges on every `regenDone`; for 100 bodies that made each regen (and each copy/paste) pay O(bodies) edge work it almost never needs. Edges are now an on-demand `bodyEdges` request the kernel serves for the live bodies, fetched via a refcount (`acquireEdges`/`releaseEdges`) only while an edge consumer — Fillet/Chamfer picking or Measure — is active, and refreshed after regens only during that window. Normal modelling regens ship meshes only.
- **Rendering tuned for a software-rasterization floor, not just a GPU.** Body material is `MeshLambertMaterial` (per-vertex lighting) rather than `MeshStandardMaterial` (per-fragment PBR), and the WebGL context runs with `antialias: false`. On a real GPU 100 lit bodies render at 60 fps either way; the change is what keeps the 100-body session ≥ 30 fps under CPU-only rasterization (the CI/sandbox path, and low-end user hardware). MSAA returns as cheaper post-process FXAA in the M7 styling pass. The perf e2e drives copy/paste to 100 bodies and asserts the render loop's peak windowed frame rate ≥ 30 fps.
- **Tree ⇄ viewport selection.** Clicking a body in the viewport (a click that isn't an orbit drag) or a tree row sets `session.selectedBodyId`; the selected body renders with an emissive highlight and is the copy source. Origin-plane visibility toggles live in the session and drive the viewport's plane groups.
**Consequences:** F8/F9/F10 are spec-conformant and covered by unit + golden tests (CopyBody preserves volume and translates its centroid; body-meta commands materialize + undo; circle-radius detection + snap candidates) and a browser e2e that builds 100 bodies via copy/paste and confirms ≥ 30 fps. Non-goals for M5 (F11 ortho/6-face views, orbit-scheme setting) remain deferred to M7.

## ADR-0017 · 2026-07-13 · Sketch preview visibility (auto-hide on consume)
**Context:** A follow-on feature request: a sketch should show as a preview before it is used, auto-hide once a feature consumes it, and be re-showable — the standard Fusion 360 sketch-visibility behavior. Until now committed sketches were only drawn while being edited (the 2D overlay); there was no persistent 3D preview and no per-sketch visibility control.
**Decision:**
- **Visibility is lazily-materialized document metadata, mirroring `bodyMeta` (F8) — NOT a field on the `Sketch` model.** A new `DocumentState.sketchMeta` holds `{ id, visible }` per sketch; a sketch with no entry reports `defaultSketchMeta` (visible). This keeps the constraint-ready `Sketch` model (and its XML codec + validation + round-trip fixtures) untouched — visibility is session-facing display state, not geometry — and reuses the exact pattern already proven for body visibility. Like `bodyMeta`, `sketchMeta` persistence rides the whole-document codec that arrives with M6; there is no per-sketch XML attribute now.
- **`SetSketchVisible` command, undoable whole-list `SketchMetaPatch`.** The browser tree gains a per-sketch visibility checkbox (parity with body rows) dispatching through the one write path (R1); the inverse patch restores the prior list exactly (R2).
- **Auto-hide fires once, on the first consuming `AddOp`, bundled into that op's transaction.** When a timeline op whose `dependencies.consumesSketch` names a sketch is added AND that sketch is currently visible AND the op is not the sketch's own producer (`producesSketch !== consumesSketch` — the Sketch op "consumes" itself), the AddOp transaction carries an extra `SketchMetaPatch` flipping the sketch hidden. Because it is one transaction, a single undo restores both the feature and the preview. It keys on visibility, not history, so editing the consuming op later never re-hides a sketch the user chose to bring back (proven in tests); adding a brand-new feature from a re-shown sketch hides it again, matching Fusion.
- **Preview drawn as 3D reference lines in the viewport, owned by `viewport/`.** The app evaluates each visible, not-currently-edited origin-plane sketch to sketch-local polylines (`evaluateSketch` → `sampleCurve`, circles closed) and passes them as a `SketchPreview { sketchId, plane, polylines }`; the viewport maps them onto the plane via `planeToWorld` and builds teal `THREE.Line`s in a dedicated group, disposed on rebuild (R8-style). The active sketch is excluded — the 2D overlay already draws it — and `viewport/` never imports `document/` (the preview payload is plain `core` geometry), preserving the depcheck viewport scope.
**Consequences:** Sketches read like Fusion: visible until used, auto-hidden on the extrude/revolve that consumes them, toggled back from the tree. Covered by unit tests (default visible; command toggle + undo; auto-hide bundled in the AddOp transaction; re-show survives an edit; unknown-sketch rejected) and a browser e2e (preview checkbox checked after finish → unchecked after extrude → re-checkable). 147 unit tests + 7 e2e green; lint/typecheck/depcheck/format clean.

## ADR-0019 · 2026-07-16 · Revolve axis: 2D Axis tool + always-available origin axes
**Context:** Follow-on request #4: revolve needs an easy, clearly-named axis. ## ADR-0023 · 2026-07-16 · Generalize the sketch plane to an offset/oriented basis (#1b foundation)
**Context:** Body-face sketching (#1b) needs the sketch-interaction layer to work on a plane that is offset and possibly re-oriented from the world origin (a body face), not just the three origin planes. The overlay, pointer-to-plane raycast, and normal-to-plane camera were all hardwired to `planeMapping(OriginPlaneId)`, whose axes pass through the world origin.
**Decision:** `PlaneMapping` gains a world `origin`; `planeToWorld = origin + u·x + v·y` and `worldToPlane` subtracts the origin first. **Consequences:** Origin-plane sketching is provably unchanged (origin = (0,0,0) collapses every formula to the old one) — verified by the full e2e suite. ## ADR-0024 · 2026-07-16 · Worker face resolution for sketch-on-face (#1b sub-PR b)
**Context:** Sketching on a body face needs the worker to turn a picked world point into a sketch plane, since only the worker holds the BRep. Body meshes reach the viewport as one merged buffer with no per-face identity, so face identity must come from the kernel.
**Decision:** A `resolveFace` request (`bodyId` + world `point`) returns a `FacePlaneResult { origin, xAxis, yAxis, normal, fingerprint }` or null. The worker (`faceResolve.ts`) meshes the body and, per face, accumulates from the triangulation the area-weighted centroid, area, and node-normal — the same triangulation data the viewport mesh comes from, so no reliance on `GProp`/`BRepClass`. The face whose triangles lie closest to the pick wins; if it is planar (`BRepAdaptor_Surface == GeomAbs_Plane`) its centroid is the plane origin, its averaged node-normal the outward normal, and an in-plane X axis is chosen deterministically (the world axis least aligned with the normal, projected) so an axis-aligned face yields an axis-aligned basis. Non-planar/greeble picks → null (the app asks for a flat face). A `FaceFingerprint { centroid, normal, areaMm2 }` — the face analogue of `EdgeFingerprint` — rides along for future regen-time re-resolution. Plumbed through `KernelClient.resolveFace` and a `kernelStore.resolveSketchFace` wrapper; all OCCT wrappers `.delete()`d (R8).
**Consequences:** The kernel can name the planar face under any pick and return a usable sketch basis — verified by a golden test on a 20×20×10 box (top face → +Z normal, on-face origin, orthonormal u/v, 400 mm²; a side face → ±X, 200 mm²; handle counter to zero). No user-visible change yet — the face-pick UX (PlanePicker "Pick a face" → CreateSketch with FacePlaneRef) is the next sub-PR; placement uses the snapshot, with parametric fingerprint re-resolution the documented follow-up.

## ADR-0028 · 2026-07-16 · M7b perspective/orthographic projection toggle
**Context:** M7 (F11) follow-up from ADR-0027: a CAD viewport needs both projections — perspective for natural 3D navigation, orthographic for measuring and axis-aligned inspection (Fusion parity). THREE uses two distinct camera classes, and the viewport had captured a single `const camera` throughout its rAF loop, pointer handlers, raycasting, and overlay projection.
**Decision:**
- **A single `viewport/cameraRig.ts` owns projection.** `CameraRig` holds the active `PerspectiveCamera | OrthographicCamera` and exposes `setAspect`, `frameBox` (zoom-to-fit for either projection), and `toggle`/`setProjection`. The pure framing math is DOM-free and unit-tested: `halfHeightAtDistance(d, fov) = d·tan(fov/2)` and `orthoExtents(halfHeight, aspect)`.
- **Seamless toggle.** Switching perspective→ortho derives the ortho frustum half-height from the current eye→target distance (`halfHeightAtDistance`), so the visible vertical extent at the target plane is unchanged; eye position, target, up, and orientation (quaternion) are all copied across. Switching back preserves the eye, and because distance is unchanged the perspective view matches again. `zoom` resets to 1 on each swap (OrbitControls dollies ortho via `zoom`).
- **Live-camera plumbing.** The viewport's `camera` became a reassignable `let` (not a fresh `const`) so every closure sees the current camera after a toggle; the toggle handler rebinds `controls.object` and the render call. `zoomToFit`/`FitTarget` moved out of `scene.ts` into the rig (perspective path kept byte-identical; ortho sizes the frustum to the box's bounding sphere). Raycasting, `.project()`-based overlay/tolerance math, and `renderer.render` all already accept the base `THREE.Camera`, so they work unchanged for both.
- **UI:** a projection button in the viewport nav strip (same layer/pattern as zoom-to-fit and the view buttons), its label showing the active projection; App passes translated `view.projection.*` labels. Projection is pure viewport state (it never touches the document), so it lives in Viewport, not a Command.
**Consequences:** The toggle works in the live app and the swap is visually stable. Unit tests cover the framing math and the rig (toggle preserves eye/up, matches half-height, round-trips back to perspective, resize updates both projections, box framing for both); an e2e toggles projection on a live body and checks the label + that the body is untouched. ## ADR-0029 · 2026-07-16 · M7c keyboard-shortcuts help overlay
**Context:** M7 (F11) deliverable: a discoverable shortcuts overlay. **Consequences:** Users can discover every shortcut. ## ADR-0030 · 2026-07-16 · M7d first-run onboarding hint

## ADR-0032 · 2026-07-17 · MASTER RULE — every menu tool has a keyboard shortcut
**Status:** Binding project rule (requested by the product owner).
**Rule:** *Every tool/action button in every menu MUST have a keyboard shortcut, and that shortcut MUST be listed in the shortcuts catalog (`app/features/help/shortcuts.ts`) and shown as the button's `title` tooltip.* New tools are not "done" until their shortcut is wired, catalogued, and title-hinted. The catalog↔i18n unit test and the shortcuts overlay are the enforcement surface.
**Shortcut scheme (v1).** Modes have separate keyspaces (sketch keystrokes are owned by the sketcher), so letters may repeat across modes; within a mode chords are unique. Ctrl/Cmd chords are global.
- **General:** N New Sketch · M Measure · Ctrl+Z/Ctrl+Y undo/redo · Ctrl+C/Ctrl+V copy/paste body · `?` help.
- **Model (non-sketch):** E Extrude · V Revolve · F Fillet · H Chamfer · B Combine · D Copy Body.
- **Document:** Ctrl+S Save · Ctrl+O Open · Ctrl+E Export STL.
- **View (non-sketch):** Z Zoom to Fit · O projection toggle · 0 Home · 1–6 Front/Back/Left/Right/Top/Bottom.
- **Sketch tools:** S Select · L Line · I Axis · R Rectangle · Shift+R Rectangle (center) · C Circle · A Arc · Shift+A Arc (center) · P Point · G Polygon · X construction · Q snap.
- **Sketch input:** Tab next field · Enter confirm · Esc cancel step · Delete remove selection · F Finish Sketch.
**Implementation.** Sketch keys extend the existing `useSketcher` keydown; non-sketch model actions live in `useModelingShortcuts` (bound only when no sketch is active); Save/Open/Export bind in their owning components (DocumentIO, ExportStlButton); view/projection/zoom bind inside `Viewport` (inert while plane-locked). Every handler ignores events targeting a text field, and digit keys stay reserved for numeric input while sketching (so view digits are non-sketch-only).
**Consequences:** Every current menu button is keyboard-reachable and tooltip-hinted; the overlay lists the full map in six groups. Future tools (the "Change" tool, intersect/cut, etc.) inherit the rule. Spot-checked by an e2e (N/M + a sketch tool hotkey) on top of the catalog unit test.

## ADR-0033 · 2026-07-17 · Fix: body-face sketches show as reference geometry immediately (#6)
**Context:** A sketch drawn on a picked body face was invisible after Finish Sketch until a feature (e.g. Extrude) consumed it — unlike an origin-plane sketch, which shows as 3D reference lines right away.
**Cause:** The committed-preview builder in `App` filtered previews to `sketch.plane.kind === 'origin'` and typed `SketchPreview.plane` as an `OriginPlaneId`, so a face-plane sketch had no way to be represented and was dropped.
**Decision:**
- `SketchPreview` now carries a `SketchPlaneBasis` (`basis`) instead of an `OriginPlaneId` (`plane`); the viewport maps preview polylines with `mappingFromBasis`, the same path the active-sketch overlay already uses — so origin and face planes render identically.
- The `Sketch → SketchPlaneBasis` helper (`originPlaneBasis` for origin, cross-product-derived normal from the face `planeSnapshot`) was extracted from `useSketcher` into `app/features/sketcher/planeBasis.ts` and shared, removing the duplicate.
- The preview computation moved out of an inline `App` `useMemo` into a pure, tested `buildSketchPreviews(sketches, sketchMeta, activeId)` in `app/features/sketcher/sketchPreviews.ts` (still excludes the active + hidden sketches).
**Consequences:** Face sketches appear as teal reference geometry the moment they're finished, matching origin-plane behavior and Fusion. Unit-tested (`buildSketchPreviews` includes a face sketch with a face basis, excludes active/hidden); the sketch-on-face and sketch-visibility e2e stay green. Layer rule respected — the basis helper lives in the app layer (it bridges document + viewport), not in the sketch layer.
